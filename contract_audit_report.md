# 專案合約與附件審查報告 (MTS-2026-001)

本報告由資深專案經理（PM）針對主合約《軟體開發服務合約》v1.0 以及附件 A、B、C 進行的深度交叉檢查與風險評估。整體而言，合約架構採用了「三軌道驗收」與「合作情境差別定價」，在機制設計上有其創意，但在**時程邏輯、合約條款與商業利益保護**上存在多處嚴重的**矛盾與高風險漏洞**。

您可以使用本文件中的對比表進行逐項審查，並在「確認」欄位以 `[x]` 標記，或在「備註」欄位記錄調整細節。

## 一、 核心矛盾與不一致清單

### 1. Apple 帳號權限與 Landing Page 開發的「先後順序悖論」
*   **矛盾描述**：
    *   [appendix-b.md](file:///Users/dennis.huang/Project/Motise/appendix-b.md#L48) 將 **2026/06/13 (M1)** 定為專案正式啟動，且前提條件包括客戶需提供「Apple Developer 組織帳號啟用與存取權限」，否則時程等比例順延。
    *   然而，[appendix-b.md](file:///Users/dennis.huang/Project/Motise/appendix-b.md#L29-L40) 又指出，作為 Apple 組織帳號申請前提的 Landing Page，是在「第一期款項到帳後 3 個工作天內」完成。若以 6/9 簽約、6/10 付款、6/13 完成 Landing Page 計算，客戶最快也只能在 6/13 之後向 Apple 提交 D-U-N-S 編號與組織帳號申請。
    *   D-U-N-S 核發與 Apple 審核通常需要 **5 至 30 個工作天**。客戶絕不可能在 6/13 (M1) 當天提供已啟用的 Apple 組織帳號。
*   **PM 評估影響**：此設定將導致專案在啟動首日即自動進入「等比例順延」狀態，合約時程流於形式。
*   **修改建議**：將「Apple 帳號權限到位」移出軌道 A (啟動) 的前置條件，改為**軌道 C (7/7 送審前) 的前置條件**。

### 2. 推播通知功能之開發、權限與驗收軌道錯置
*   **矛盾描述**：
    *   **開發與驗收不一致**：[appendix-b.md](file:///Users/dennis.huang/Project/Motise/appendix-b.md#B-3) 將推播通知（M-06）列在**軌道 B**（前端與後台，6/23-7/4）開發；但 [appendix-c.md](file:///Users/dennis.huang/Project/Motise/appendix-c.md#C-5) 卻將「課程提醒推播（C-03）」與「搶購提醒推播（C-04）」列在**軌道 C** 驗收。
    *   **權限提供時機矛盾**：[appendix-a.md](file:///Users/dennis.huang/Project/Motise/appendix-a.md#L182) 規定 Firebase 專案存取權限於「上線前」提供。然而，若推播功能在軌道 B (6/23-7/4) 開發，開發方在開發期間就必須進行 Firebase 整合測試，不可能等到「上線前（軌道 C）」才拿到權限。
    *   **後端時程遺漏**：推播通知需要後端排程（如 24 小時與 1 小時提醒）與 API 支持，但 [appendix-b.md](file:///Users/dennis.huang/Project/Motise/appendix-b.md#B-3) 的軌道 A（後端，6/13-6/27）工作項目中完全沒有提及推播相關後端開發。
*   **PM 評估影響**：開發與驗收軌道脫節，且缺乏開發期權限，會導致軌道 B 無法順利完成或無法在軌道 B 驗收該功能，拖延付款時程。
*   **修改建議**：
    1. 將 Firebase 權限提供時間提前至軌道 B 啟動前（6/23）。
    2. 將推播通知的後端 API 開發明確納入軌道 A，前端與 FCM 整合納入軌道 B，並統一在軌道 B 進行第一階段功能驗收，軌道 C 僅驗收正式環境下的推播表現。

### 3. 長期合作確認時點與交付時程的邏輯衝突
*   **矛盾描述**：
    *   [contract-zh-v3.md](file:///Users/dennis.huang/Project/Motise/contract-zh-v3.md#L108) 要求客戶於 **2026/07/25 前**書面確認長期合作（情境一）或不合作（情境二），逾期自動以情境二（高價版 $2,499）計算。
    *   然而，根據 [appendix-b.md](file:///Users/dennis.huang/Project/Motise/appendix-b.md#L46)，系統目標上線日為 **7/22**，正式交付日為 **8/4**，軌道 C 驗收期為上線後 7 個工作天（約至 **7/31**）。
*   **PM 評估影響**：在 7/25 時，系統可能剛上線 3 天（甚至可能因 App Store 審核延遲而尚未上線），軌道 C 尚未驗收完畢，程式碼與技術文件也尚未移交。客戶在無法驗證系統穩定性與程式碼品質的情況下，被迫做決定。這會引發客戶反彈，或導致客戶被迫申請延期至 8/4。
*   **修改建議**：將長期合作的確認期限調整為**軌道 C 驗收通過後 5 個工作日內**，或者與正式交付/尾款支付時點綁定。

### 4. 日期與星期標記錯誤
*   **錯誤描述**：[appendix-b.md](file:///Users/dennis.huang/Project/Motise/appendix-b.md#B-2) 中的多處里程碑日期與星期不符（以 2026 年日曆為準）：
    *   M1: 2026/06/13 標記為 (五)，實際為 **星期六**
    *   M2: 2026/06/18 標記為 (三)，實際為 **星期四**
    *   M3: 2026/06/20 標記為 (五)，實際為 **星期六**
    *   M4: 2026/06/27 標記為 (五)，實際為 **星期六**
    *   M5: 2026/07/04 標記為 (五)，實際為 **星期六**
*   **PM 評估影響**：星期標記錯誤會導致「工作天」與「日曆天」計算混淆，且多個關鍵里程碑設在週六，實務上可能影響雙方配合意願。
*   **修改建議**：修正星期標記，並將里程碑儘量調整至工作日（週一至週五）。

## 二、 商業與法律風險分析

### 1. 情境三的計費漏洞（引導客戶不予驗收的道德風險）
*   **條款漏洞**：
    *   [contract-zh-v3.md](file:///Users/dennis.huang/Project/Motise/contract-zh-v3.md#L105) 規定，若進入「情境三（未能全數通過驗收或嚴重延誤）」，費用以 **$1,800 USD** 計算，並依第六條扣減賠償。
    *   若客戶不打算長期合作（情境二，總價 **$2,499 USD**），只要挑出系統的微小瑕疵，主張「未能全數通過驗收」，合約計費基準就會立刻降為 **$1,800 USD**。
*   **PM 評估風險**：這對開發方而言是極大的商業風險。合約機制非但沒有鼓勵客戶協助完成驗收，反而**提供了 $699 USD 的財務誘因，誘導客戶故意不讓專案完美結案**。
*   **修改建議**：情境二與情境三的基準費用應統一（例如均為 $2,499 USD）。若因開發方原因導致部分功能無法交付，應以該功能之比例進行扣款，而非直接降低整體合約的價格基準。

### 2. 開發方智慧財產權與系統控制權的失衡
*   **條款漏洞**：
    *   [contract-zh-v3.md](file:///Users/dennis.huang/Project/Motise/contract-zh-v3.md#L126) 規定 GitHub Repository 於客戶帳號下建立，開發方僅為 Collaborator。
    *   [contract-zh-v3.md](file:///Users/dennis.huang/Project/Motise/contract-zh-v3.md#L128) 規定 AWS Production 環境存取權限於**軌道 C 驗收通過後即提供客戶**，不受第三期尾款移交時序限制。
*   **PM 評估風險**：
    *   代碼存放在客戶帳號下，客戶可隨時移除開發方的 Collaborator 權限，無償獲取已開發的代碼。
    *   在客戶付清第三期尾款前，就已經拿到了 AWS Production 權限與 GitHub 程式碼控制權。開發方將失去所有制約手段，面臨**拿不到尾款的極高風險**。
*   **修改建議**：
    *   開發期間 GitHub Repository 應建立在開發方帳號下（或雙方共同託管的獨立組織），尾款結清後移交 ownership。
    *   AWS Production 環境權限移交，必須與第三期尾款支付同時（或於款項到帳後 3 個工作天內）執行。

### 3. Stripe KYC 啟動時限過於苛刻
*   **條款漏洞**：[contract-zh-v3.md](file:///Users/dennis.huang/Project/Motise/contract-zh-v3.md#L67) 要求客戶於 **2026/06/10** 前啟動 Stripe KYC。
*   **PM 評估風險**：合約簽署日期為 6/9，若電子簽章流程稍有延遲（例如 Ding Ding 在 6/10 才完成簽署），客戶在合約生效當天或甚至生效前就面臨違約風險。這是不合理的時程壓迫。
*   **修改建議**：改為「合約生效後 3 個工作天內」或「第一期款項支付後 2 個工作天內」。

### 4. 延遲賠償公式的實用性低且定義模糊
*   **條款漏洞**：
    *   [contract-zh-v3.md](file:///Users/dennis.huang/Project/Motise/contract-zh-v3.md#L138) 賠償公式：`已支付款項 × (未完成驗收項目數 ÷ 總驗收項目數) × 50%`。
    *   合約未定義何謂「項目未完成」（是有 Bug 算未完成，還是完全未開發？）。
    *   以總項目 43 項計算，單一項目未完成的賠償金額極低（如僅約十幾美元），且核心功能（如 Stripe 付款）與非核心功能（如 Landing Page 顏色）的權重完全相同，不具實務操作性。
*   **修改建議**：取消公式計算，改為「若因開發方原因導致關鍵里程碑延誤超過 14 日，每日扣除總合約金額之固定比例（如 0.5%），上限為 20%」；或明確定義何為「關鍵功能（如付款、預約、掃碼）」，僅針對關鍵功能未交付進行比例扣款。
