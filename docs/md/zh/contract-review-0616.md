## 1. 核心業務邏輯矛盾 (PM & 技術角度)

### 1.1 課包取消與過期機制的「邏輯死循環」
* **文件位置：** [附件 A 第 A-3-2 節](file:///Users/dennis.huang/Project/motise-docs/docs/md/zh/appendix-a.md#L56-L60)、[附件 A 第 A-3-11 節](file:///Users/dennis.huang/Project/motise-docs/docs/md/zh/appendix-a.md#L156-L160)
* **矛盾說明：**
  * **規則一：** 系統設有「防呆機制」，若「目標課程開始時間」晚於所扣抵課包之到期日（`expired_at`），系統會攔截並禁止預約（即預約成功的前提是：課程開始時間 $T_S \le$ 課包過期時間 $T_E$）。
  * **規則二：** 課程取消政策規定，學員必須在「課程開始前 24 小時以上」取消（即取消時間 $T_C \le$ 課程開始時間 $T_S - 24$ 小時）。
  * **規則三：** 兩節均規定「若退回時原課包已逾期，退回之堂數維持失效狀態」。
  * **邏輯衝突：** 
    結合上述公式，當學員進行「合規取消」時，其時間關係為：
    $$T_C \le T_S - 24 \text{ 小時} < T_S \le T_E$$
    這代表**合規取消的時間點 $T_C$ 必定早於課包過期時間 $T_E$**。既然取消時課包根本還沒過期，系統即時退回堂數時，該課包在退回當下**絕對是 ACTIVE 狀態，不可能已逾期**。
    因此，「退回時原課包已逾期，退回之堂數維持失效」是一個**在邏輯上永遠無法被觸發的無效條款**。
* **潛在風險：** 此條款唯一可能觸發的情境是「因開發方或教師因故取消課程，系統被動退回堂數給學員時，原課包已過期」。但合約並未定義「非學員因素取消」之退回規則，將導致平台與消費者間的嚴重糾紛。
* **修訂建議：** 
  1. 刪除該死邏輯文字，或明確限制該條款僅適用於「因不可抗力或教師/平台端主動取消課程，而退回堂數至已過期課包」之情境。
  2. 明確約定「若因教師或平台方取消課程，退回之堂數應無條件展延 30 天或以其他方式補償」，以符合美國各州消費者保護法。

### 1.2 取消退回堂數之「再次預約限制」與技術選型的矛盾
* **文件位置：** [附件 A 第 A-3-2 節](file:///Users/dennis.huang/Project/motise-docs/docs/md/zh/appendix-a.md#L60) 規定：「由取消退回之堂數再次預約之課程，不得再次取消。」
* **矛盾說明：**
  * 在 [附件 A 第 A-4-3 節（核心資料表）](file:///Users/dennis.huang/Project/motise-docs/docs/md/zh/appendix-a.md#L222) 中，`user_pass_packages` 的結構為「數量制」（欄位僅有：總堂數、剩餘堂數、狀態等）。
  * 當一筆退回的堂數進入該課包實例，它會與該課包內其他「尚未使用的堂數」混合。在目前的資料庫結構與 API 設計下，系統**無法區分並追蹤**哪一堂是「原本的堂數」，哪一堂是「取消退回的堂數」。
* **技術風險：** 若要強制實現此業務規則，必須將資料結構重構為「單一堂數明細表」（例如新增 `user_passes` 資料表以追蹤每一堂 Pass 的狀態歷史與來源），這會大幅增加開發工時、資料庫複雜度與系統負載，且與目前 A-4-3 的技術選型矛盾。
* **修訂建議：** 從 PM 角度出發，建議評估此限制之商業價值。若非絕對必要，建議刪除「退回堂數不得再次取消」之限制，直接併入每月最多取消 3 次之總量管制即可，以簡化系統複雜度。

## 2. 法律與財務風險條款 (法律顧問角度)

### 2.1 任意終止結算公式的「分母歧義」與「權重失衡」
* **文件位置：** [主合約第 5.9 條](file:///Users/dennis.huang/Project/motise-docs/docs/md/zh/contract-zh.md#L186-L196) 任意終止公式：
  $$\text{終止日應付金額} = \text{適用情境總費用} \times \left( \frac{\text{截至終止日已通過驗收之項目數}}{\text{附件 C 總驗收項目數}} \right) - \text{已支付款項}$$
* **矛盾說明：**
  1. **分母定義歧義：** 主合約並未寫明「總驗收項目數」的具體數字。雖然 [附件 C 第 C-6 節](file:///Users/dennis.huang/Project/motise-docs/docs/md/zh/appendix-c.md#L192-L201) 寫有「合計 52」，但這包含了 3 項「前置工作」。若前置工作已於早期鎖定，分母是否固定為 52？若後續因範疇變更（SOW）增加或減少項目，此分母是否會變動？定義不明易生爭議。
  2. **等權重計算的財務風險：** 公式將「功能大小」與「工時價值」完全不同的項目給予了相同的權重。例如：B-03「手機號碼綁定」（輕量 API 串接）與 A-12「課包購買與堂數扣抵 API」（後端核心交易邏輯）皆佔 $1/52$。
* **法律風險：** 
  * **對開發方：** 若合約在開發方完成所有複雜的後端架構（軌道 A）但前端（軌道 B）尚未全面開工時終止，開發方付出了 70% 的核心心力，卻只能依項目數比例（如 16/52 = 30.7%）結算，會蒙受巨大損失。
  * **對客戶方：** 若計算結果為負數，主合約規定「開發方無需退還已收款項」。如果客戶已支付第二期款（60%），但通過項目數極少，客戶將無法追回多付的款項。
* **修訂建議：** 建議廢除「按項目數平分」的結算方式，改採「按已完成之階段（軌道）所占該情境之付款比例，並針對未完成軌道之工時進行協議結算」或「針對前置工作、軌道 A、B、C 各自約定明確的結算權重百分比（例如前置工作 10%、軌道 A 30%、軌道 B 35%、軌道 C 25%）」。

### 2.2 長期合作框架協議（附件 D）之「情境確認期限」缺失與術語混亂
* **文件位置：** [主合約第 4.2 條](file:///Users/dennis.huang/Project/motise-docs/docs/md/zh/contract-zh.md#L121)、[主合約第 4.3 條](file:///Users/dennis.huang/Project/motise-docs/docs/md/zh/contract-zh.md#L123)、[附件 D 第 D-8.2 條](file:///Users/dennis.huang/Project/motise-docs/docs/md/zh/appendix-d.md#L232)
* **矛盾說明：**
  * **術語不一致：** 主合約第 4.2 條與附件 D 第 D-8.2 條規定，附件 D 須於「主合約完成雙方電子簽署之**同時**」完成「**書面**簽署」。但主合約第 4.3 條卻指稱「確認長期合作」是指在「**情境確認期限前**完成書面簽署」。
  * **定義缺失：** **主合約與各附件中，完全沒有定義何謂「情境確認期限」**。這是一個嚴重的法律漏洞。若雙方在簽署主合約後數天內才簽署附件 D，此時究竟適用情境一（折扣價）還是情境二（原價）？
  * **簽署方式矛盾：** 條款要求「書面簽署」，但 [附件 D 第 9.1 條](file:///Users/dennis.huang/Project/motise-docs/docs/md/zh/appendix-d.md#L249-L265) 卻規定「本附件採電子簽名方式完成，透過 Dropbox Sign 平台進行」。
* **修訂建議：** 
  1. 刪除「情境確認期限」等模糊字眼。
  2. 統一術語，將「書面簽署」明確定義為「書面簽署（含透過 Dropbox Sign 等平台進行之電子簽署）」。
  3. 明確約定：「若雙方於簽署主合約之日起 [X] 個日曆天內（或於主合約簽署之同時），完成附件 D 之電子簽署，即適用情境一，否則自動適用情境二。」

### 2.3 免費保固與付費維護的界線重疊與模糊
* **文件位置：** [主合約第 4.7.3 條](file:///Users/dennis.huang/Project/motise-docs/docs/md/zh/contract-zh.md#L141-L143)、[附件 D 第 D-5.6.1 節](file:///Users/dennis.huang/Project/motise-docs/docs/md/zh/appendix-d.md#L135-L150)
* **矛盾說明：**
  * 主合約第 4.7.3 條將「免費保固」限縮在極窄的範圍：「僅限於尾款支付後 60 日內，App 因原始程式碼缺陷遭平台**功能性拒絕（Functional Rejection）**時，開發方有義務免費修正至審查通過為止。」
  * 該條同時規定：「正式交付驗收完成後發現之此類問題（核心功能 Bug 或閃退），依附件 D 第 D-5.6 節維護服務辦理，不適用免費保固。」
* **商業與法律風險（對客戶）：** 這意味著只要 App 成功上架或通過實機驗收並支付尾款，**即使在 60 天內發現了 1.0 範疇內的核心功能重大 Bug（例如 Stripe 漏單、資料庫並發鎖死等），開發方也完全沒有免費修復的義務**。客戶必須每月支付 USD $500 的維運費（SOW）或另行付費，開發方才予修復。這與一般軟體開發合約中「保固期內免費修復符合規格之瑕疵（Bug）」之商業慣例嚴重相左。
* **修訂建議：** 
  明確區分「瑕疵擔保（保固）」與「系統維運（維護）」：
  1. **保固：** 尾款支付後 60 天內，凡屬 v1.0 開發規格內之功能，若出現與規格不符之 Bug、閃退或異常，開發方應免費修復。
  2. **維護（附件 D）：** 針對流量增長之優化、第三方套件相容性升級、不屬 v1.0 規格之微調等，方屬付費維護範疇。

### 2.4 附件 D 「費率上限」定義模糊
* **文件位置：** [附件 D 第 D-5.4 節](file:///Users/dennis.huang/Project/motise-docs/docs/md/zh/appendix-d.md#L113-L124)、[附件 D 第 D-3.2 節](file:///Users/dennis.huang/Project/motise-docs/docs/md/zh/appendix-d.md#L54-L62)
* **矛盾說明：**
  * D-5.4 將「費率上限基準」定義為一個**區間**：「以 **USD 50–60 / 小時** 為費率上限基準（以下簡稱『費率上限』）」。
  * D-3.2 規定：「若開發方所提報酬報價『超出 D-5.4 所定費率上限』而致無法達成共識，客戶無須支付當期差額補足款項。」
* **法律爭議：** 由於費率上限是一個區間（50–60），法律上「超出上限」的判定基準將極具爭議。若開發方報價為 USD 55/小時，因其在區間內，但超出了區間的下限（50），這是否算「超出費率上限」？
* **修訂建議：** 將費率上限修改為單一確切數字，例如：「以 **USD 60 / 小時**（或日薪 USD 480）為費率上限基準」。

## 3. 專案時程與管理衝突 (Senior PM 角度)

### 3.1 軌道 C 驗收期起算日與尾款付款時機之潛在爭議
* **文件位置：** [主合約第 4.7 條](file:///Users/dennis.huang/Project/motise-docs/docs/md/zh/contract-zh.md#L131)、[主合約第 4.7.2 條](file:///Users/dennis.huang/Project/motise-docs/docs/md/zh/contract-zh.md#L135-L139)、[附件 C 第 C-1-1 節](file:///Users/dennis.huang/Project/motise-docs/docs/md/zh/appendix-c.md#L22)
* **矛盾與實務疑慮：**
  * [附件 C 第 C-1-1 節] 規定軌道 C 的開發期間為 `7/30 – 8/31`，驗收期為「起算日起 14 個工作天」。
  * 依 [主合約第 4.7 條]，起算日為「開發方將 App 提交至 TestFlight...或平台審核通過之日（孰先發生者）」。若開發方於 8/02 提交 TestFlight，則 14 個工作天的驗收期大約在 8/21 屆滿。
  * 然而，[主合約第 4.7.2 條] 規定，若「App 於雙平台均完成正式上架」或「召開線上驗收會議實機功能測試通過」，即為尾款付款指示之起算日。
  * **實務爭議：** 若 App 於 8/14 成功上架，依合約應立即付清尾款。此時 14 個工作天的驗收期尚未屆滿。客戶可能會主張「App 雖上架，但我仍有權在 14 個工作天驗收期內進行測試並提出修改」。
* **修訂建議：** 
  在主合約中明確註記：「App 於雙平台正式上架之日，即視為軌道 C 驗收合格通過，14 個工作天之驗收期自動提前終止，客戶應依第 4.1 條約定支付尾款。」以排除客戶事後要求繼續使用 14 天驗收期之主張。

### 3.2 附件 D 附表一之「免費維護期」與最低發案義務之排他性
* **文件位置：** [附件 D 附表一 第 S1-3.4 節](file:///Users/dennis.huang/Project/motise-docs/docs/md/zh/appendix-d.md#L323-L324) 規定：「本 SOW 包含最後一個里程碑驗收通過後 3 個月之基本維護，維護範圍依第 D-5.6 條辦理。」
* **PM 財務規劃疑慮：**
  * 附件 D 第一週期為 `2026/08/31 – 2026/12/01`。
  * 若此 SOW 於 2026 年 10 月底完成（M3 驗收），則 3 個月的免費維護期將涵蓋 `2026/11 – 2027/01`（跨越到第二週期）。
  * 雖然在此維護期內客戶不需額外支付每月 USD $500 的維護費，但這會導致第二週期（`2026/12/01 – 2027/03/03`）開始時，系統仍處於「前案免費維護」狀態。
  * 依 [附件 D 第 D-3.1 條]，客戶在第二週期仍必須「簽署一份新的 SOW」，否則就必須支付 USD $175 的差額補足款。
* **修訂建議：** 
  PM 應提醒客戶，第一案贈送的 3 個月維護期，**無法抵免**後續週期的「最低發案義務」。第二週期不論系統是否在維護期內，客戶仍須提出新需求（或簽署其他付費維護 SOW）以滿足 D-3 條款。

### 4.2 工作天與日曆天定義
* [主合約第 8.7 條](file:///Users/dennis.huang/Project/motise-docs/docs/md/zh/contract-zh.md#L257) 明確定義了「工作天」（台灣銀行營業日）與「日曆天」（連續日曆日）。
* 惟在時程管理上，各條款交替使用兩者（例如：催告 3 個工作天、付款逾期 7 個/21 個日曆天、差額付款 10 個工作天、終止通知 21 個日曆天）。
* **PM 提醒：** 雖然定義清晰，但實務執行（如計算逾期利息或終止日）極易因跨越台灣與美國的國定假日而產生算日爭議，PM 在後續專案進度管理中須建立專屬的「合約日曆對照表」以防計算錯誤。
