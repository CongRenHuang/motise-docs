## 二、本輪已修補（3 項）

### #1 軌道 C「驗收期起算點」三處定義打架 — 已修
- **問題：** 主約 4.7 起算 =「交付測試或審核通過之日」；附件 C C-1-1／C-5 起算 =「上線後」；4.7.1 緩衝 =「提交審查之日」。三個錨點時間先後不同，客戶可挑對己有利者拖延「視同通過」與尾款觸發。
- **修補：** 附件 C C-1-1 表格與 C-5「驗收期」改為「依主合約第 4.7 條所定起算日起算 14 個工作天」，並註明「正式上線」非唯一起算點。中英＋HTML 同步。

### #2 「工作天」無定義、且與「工作日」混用 — 已修
- **問題：** 全約大量用「工作天／工作日」，但從未定義依何方曆法、是否排除假日；台美假日不同，客戶可挑曆法解釋。
- **修補：**
  - 主約新增 **8.7 期間定義**：「工作天」= 中華民國（台灣）銀行營業日，不含週六日及台灣國定假日；「日曆天」= 連續日曆日含假日；「工作日」與「工作天」同義。
  - 全文「工作日」統一改為「工作天」（contract-zh、appendix-a/b/c）。
  - EN 同步新增 contract-en 8.7 Definition of Time Periods。

### #5 附件 D $175 雙重收取風險 — 已修
- **問題：** 「終止日前已逾期未簽」之週期，可能同時落入 D-3.2（截止後付 $175 差額補足）與 D-7.3（終止時付 $175 返還款），同一週期被收兩次。
- **修補：** D-7.3 加一句「同一週期若已依 D-3.2 支付差額補足款，不再依本款重複計收；已付差額補足款視為已履行本款返還義務」。中英＋HTML 同步。

---

### #6 5.9.1 加速終止 與 6.1 延遲賠償 是否併用未明 — 已修
- **問題：** 開發方遲延時客戶可能同時主張 5.9.1 終止（按公式少付）+ 6.1 賠償（已付款 20%）。兩者關係未寫。
- **修補：** 主約新增 **6.4 與終止條款之關係**——同一遲延事由，5.9／5.9.1 終止與 6.1 賠償**擇一行使**，不得併行。中英＋HTML 同步。

### #7 交付日(8/11) vs 驗收期屆滿(~8/13) 重疊 — 已修
- **問題：** 上線目標 7/25 + 14 工作天 ≈ 8/13，晚於交付目標 8/11，客戶可質疑「交付日已過、驗收未完=逾期」。
- **修補：** 主約 **2.1** 加註「目標交付日為里程碑規劃目標，軌道 C 驗收期依 4.7 獨立計算，得跨越該目標日，驗收與修正程序不構成開發方逾期」。中英＋HTML 同步。

## 三、記錄待辦（談判讓步結構，本輪不動）

### #3 終止公式 vs 工作量錯配（結構性）
5.9 公式按「已通過項目數 ÷ 46」均權，但付款 30/30/40。軌道 C 僅佔 11/46≈24% 項數，卻含送審／上線／正式環境等最重工作。客戶於 App 送審中（C-01/C-02 未「通過」）援引 5.9 任意終止 → 只付約 76% 項數卻已取得約 95% 成果。
**緩解現況：** 4.7.2 視同通過兜底 + 14 天通知期。**未來建議：** 軌道 C 上架項加權，或加「已提交雙平台審查後終止，軌道 C 視為已完成過半」擬制條款。

### #4 5.4.2 緊急例外「誤判無罰則」可被武器化
客戶憑合理懷疑 + 證據即可暫停開發方 Admin 權限，誤判僅順延、雙方無罰則。零成本 → 可反覆假性觸發拖垮節奏。此為 6/13 談判讓步結果（誤判無罰則為客戶要求底線），暫不動。
**未來建議：** 加「同一專案誤判達 2 次後，再經認定誤判者視為客戶依 5.9 提前終止」之嚇阻上限。
