## 第三條｜驗收機制

3.1 本系統驗收依附件 C《驗收項目文件》分三個軌道進行，各軌道驗收標準以附件 C 之記載為準。

3.2 各軌道驗收期屆滿，客戶方未提交書面驗收結果者，開發方應發出催告通知；催告送達後逾 3 個工作天，客戶方仍未提交書面驗收結果者，視為該軌道全數通過驗收。

3.3 每週同步會議後 48 小時內，客戶方未提出書面異議，視同確認該階段進度，不得於後續驗收時重新提出。

3.4 各軌道一旦驗收通過，該軌道項目即鎖定，不得於後續軌道或正式交付時重新提出異議。

3.5 驗收期間提出之新需求，不列入本次驗收範疇，另行評估報價後進入下一版本開發。

3.6 **主觀退件無效：** 客戶方提出之驗收不通過或退件意見，須具備明確之功能性問題說明或與雙方確認之設計規格不符之具體事證。以主觀性描述（如「感覺不對」、「體驗不順暢」等）作為退件理由者，不生退件之效力，仍視同通過。

3.6.1 **效能爭議之客觀判定：** 若爭議涉及系統效能（如頁面載入時間、列表滾動流暢度、API 回應時間等），且雙方於 TestFlight 或同等測試環境，依附件 A 第 A-3-10 節所載測試前提實測之數據，未達該節所定效能基準者，視為功能性問題，適用第 3.7 條複驗機制處理。未達附件 A-3-10 基準之優化要求、或無附件規格依據之主觀體驗描述，仍適用本條（3.6）主觀退件無效原則，不生退件效力；開發方就此等要求應本於誠信原則評估是否優化，但無義務執行。客戶方得於本合約簽署前，以書面提出合理之商業標準驗收項目（如特定 UI/UX 互動、動畫或視覺呈現之具體可驗證標準），經雙方討論並書面同意後，納入附件 A 規格或附件 C 驗收項目，作為前述「附件規格依據」之一部；簽署後未經此程序納入者，不構成獨立之驗收標準。

3.7 **複驗限制與時限：** 針對客戶方於驗收期內提出之書面修改意見清單，開發方完成修正後，客戶方僅能就該清單內原列項目之修正結果進行複驗。客戶方不得於複驗時提出原清單外之新修改意見或重複挑剔，複驗時間以 3 個工作天為限；複驗期限屆滿，客戶方未提交複驗結果者，開發方應發出催告通知，催告送達後逾 3 個工作天客戶方仍未提交複驗結果者，視為該清單全數複驗通過。

3.7.1 **回歸缺陷（Regression Bug）：** 若開發方為修正本次清單項目所為之程式碼變更，直接導致同一驗收清單內其他項目、或先前已通過驗收項目之既有功能產生新的功能性錯誤（須符合第 3.6 條標準，具備明確功能性問題說明），客戶方得於複驗期間一併提出，不受第 3.7 條「不得提出清單外新意見」之限制，亦不構成違反第 3.4 條軌道鎖定。開發方應一併修正；因此所需之額外修正時間，複驗期限得就該項目順延 3 個工作天，其餘項目複驗期限不受影響。回歸缺陷之複驗僅限確認該缺陷已修正，不得再衍生提出其他新意見。

3.8 **每週同步會議：** 雙方原則上於每週五 24:00 至翌日（週六）01:00（台灣時間，UTC+8；對應美國太平洋時間週五上午 09:00–10:00）進行線上同步會議，同步專案進度。如遇雙方無法於該時段進行會議，應提前告知並協調改期，但仍應於該週內完成一次同步會議。
