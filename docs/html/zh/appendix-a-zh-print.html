A-3 v1.0 功能細項

A-3-1 帳號與角色

  • 支援三種登入方式：Email / 密碼、Google OAuth、Apple Sign-In（含 Face ID / Touch ID）
  • 支援手機號碼獨立綁定（非登入方式，為帳號附加資訊）
  • 系統角色分三級：Student（學員）、Teacher（教師）、Admin（管理員）
  • 各角色 API 存取權限獨立，跨角色存取返回 403

A-3-2 課程功能

  • 課程列表顯示：名稱、時間、價格、剩餘名額；下架課程不顯示
  • 課程詳情頁：完整說明、地點、名額、講師姓名及個人介紹
  • 名額鎖定：名額更新採資料庫原子操作，確保並發預約不超額，滿額後預約按鈕關閉
  • 預約截止時限：超過截止時間後預約按鈕自動關閉
  • 限時搶購（Flash Sale）：課程可設定開賣時間與結束時間，僅於該時間區間內可預約；額滿後預約按鈕關閉，與一般課程共用名額鎖定機制
  • 名額即時鎖定與釋出：使用者以課包堂數預約課程時，系統即時以原子操作鎖定名額並扣抵 1 堂（詳見 A-3-11）；未持有效堂數者無法預約。
  • 名額僅於使用者依取消政策（課程開始前 24 小時以上）取消時釋出，並同時依取消政策退回堂數至原課包。
  • 防超賣與資料一致性：名額異動（扣減、釋出）以資料庫交易層級鎖定/原子操作為基礎保證，確保顯示與實際庫存一致。
  • 取消政策（堂數退回機制，採堂數制）：
    • 學員取消： 課程開始前 24 小時以上得取消，系統自動釋出名額，不退款，將該次預約所扣抵之 1 堂堂數退回原扣抵之課包（詳見 A-3-11）；課程開始前 24 小時內不可取消。學員合規取消之時點必早於課包到期，退回之堂數回到原課包後仍為有效。
    • 課程被取消（教師下架／平台端）： 經後台下架或取消「已有學員預約」之課程時，系統將受影響學員已扣抵之 1 堂堂數退回原扣抵之課包，沿用原課包之到期時間（expired_at）；若退回時原課包已逾期，退回之堂數維持失效（EXPIRED）狀態，不展延、不另補發。
    • 取消次數限制：每一帳號每月（30 天）以「24 小時以上之取消」最多 3 次為限。
  • 我的課程頁：已預約、已購買、課包與剩餘堂數、歷史紀錄分類

A-3-3 付款流程

  • 透過 Stripe Payment Intent 建立付款請求
  • 訂單狀態以 Stripe Webhook 為唯一更新依據，並對 Webhook 事件進行冪等處理：使用者完成課包購買付款後即使立即斷線，後端仍能正確更新訂單狀態，並將購得堂數正確寫入對應之課包實例（user_pass_packages）；使用者重新連線後，於「我的課包」可正常顯示已入帳之堂數
  • 名額扣減與 QR Code 生成屬「以堂數預約課程」之即時流程（見 A-3-2／A-3-11），於預約成功時原子完成，非由付款 Webhook 觸發
  • 付款失敗顯示明確錯誤訊息，堂數不入帳、課包不生成
  • 已預約之課程不再顯示預約按鈕，防止重複預約（重複扣抵堂數）；同一筆課包購買訂單以冪等處理防止重複扣款
  • 購買課包前顯示取消政策（含 30 天到期失效）條款文字，使用者須勾選同意後方可付款
  • 課包（Class Pass）購買亦透過本節 Stripe 付款機制與冪等處理辦理；本系統採「先購買課包取得堂數、再以堂數預約課程」之主要購課流程，課包購買與堂數扣抵之完整規格詳見 A-3-11

A-3-4 QR Code 簽到

  • 已預約學員可於課程卡片查看並出示專屬 QR Code
  • QR Code 與學員帳號及課程綁定，課程開始後 15 分鐘自動失效
  • 教師介面提供相機掃描入口，掃描後顯示學員姓名、課程資訊及出席狀態
  • 已核銷 QR Code 再次掃描顯示「已簽到」，不重複計算
  • 掃描無效或已失效 QR Code 顯示明確錯誤訊息

A-3-5 後台管理（Web-based）

  • 課程管理：新增、修改、上架、下架；上架後前台即時顯示。下架／取消「已有學員預約」之課程時，依 A-3-2 將受影響學員之堂數退回原課包（過期則維持失效、不展延）
  • 會員管理：查詢會員、指派或移除教師角色、停用帳號
  • 訂單查詢：學員姓名、課程名稱、付款狀態、金額
  • 出席記錄：各課程出席名單與簽到狀態
  • 課包管理：查詢學員課包與剩餘堂數、到期狀態（唯讀查詢）。手動延長到期日／手動調整堂數及其稽核紀錄（Audit Log）不列入 v1.0 範疇，順延至附件 D 或後續 SOW；MVP 階段如遇特例由開發方以人工方式調整資料庫紀錄處理，依實際工時另計，不含於附件 D 第 D-5.6 節維護服務時數

A-3-6 推播通知

  • 課程提醒：已預約課程於開始前 24 小時及 1 小時各發送一次
  • 搶購提醒：使用者可對指定課程設定開搶提醒，開搶前收到推播
  • 使用者可自行設定是否開啟各類推播通知

A-3-11 課包購買與堂數管理（Class Pass 課包系統）

本節為本系統 v1.0 之主要購課入口。學員須先購買課包（Class Pass）取得堂數，再以堂數預約課程。

課包購買

  • 提供多種堂數規格之課包（如 1 堂／5 堂／10 堂），各規格售價由客戶於後台設定。
  • 課包採等值制（堂數制）：1 堂堂數可預約任一門課程，不論該課程定價；不採金額扣抵制。
  • 學員透過 Stripe 付款購買課包，付款流程與 Webhook 冪等處理依 A-3-3 辦理。
  • 結帳頁須提供強制勾選之同意條款：「本課包為限時 30 天方案，逾期未使用之堂數自動失效且不予退費」，勾選後方可付款。

課包實例與資料結構

  • 每筆購買各自生成獨立之課包實例（User_Pass_Packages），不得僅以使用者單一欄位累加堂數。
  • 課包實例欄位至少包含：total_passes（總堂數）、remaining_passes（剩餘堂數）、activated_at（購買即啟用時間）、expired_at（精確至秒之 30 天後到期時間）、status（ACTIVE / USED_UP / EXPIRED）。
  • 同一帳號得同時持有多個課包實例；課包不可轉讓予他人。

堂數扣抵（FIFO）

  • 學員預約課程時，系統自 status = ACTIVE 之課包中，優先扣抵 expired_at 最接近現在者（最快到期先扣）。
  • 預約成功扣減 1 堂；remaining_passes 歸零時，該課包 status 轉為 USED_UP。

過期判定（即時判定，不採定時任務）

  • 系統不另設背景定時任務（Cron Job）批次標記過期。課包是否過期，於「預約」與「查詢課包」時，以當下時間與 expired_at 即時比對判定；超過 expired_at 之課包不得用於預約，其剩餘堂數即時視為失效。
  • 後台課包列表之 status 於讀取（fetch）時即時反映實際過期狀態；DB 中 status 欄之背景批次更新不屬 v1.0 範疇，不影響功能正確性——過期課包於任何讀取或預約時均正確判定為失效，學員無法以過期課包預約。

預約與取消之時空防呆

  • 禁止預約過期課：預約時若「目標課程開始時間」晚於所扣抵課包之 expired_at，系統須攔截並提示。
  • 取消退回校驗：依 A-3-2 各取消情境退回之 1 堂，回到原扣抵之課包。學員合規取消（必早於到期）退回後仍為有效；因課程被下架／平台端取消之被動退堂，若原課包已過期，退回之堂數維持失效狀態、不展延（與 A-3-2 一致）。

Admin 例外調整（不列入 v1.0 範疇）

  • 後台「手動延長到期日」「手動調整堂數」功能，及其對應之 Audit Log 稽核紀錄，不列入 v1.0 開發範疇，順延至附件 D 或後續 SOW 評估。
  • MVP 階段如遇特例（如學員生病、爭議），暫由開發方以人工方式調整資料庫紀錄處理；該等人工處理依實際工時另計，不含於附件 D 第 D-5.6 節維護服務時數。

前端告知義務

  • 課包卡片須以醒目方式（高亮或紅色）顯示到期時間與剩餘天數，例如「請於 YYYY/MM/DD HH:MM 前使用完畢（剩餘 X 天）」。

⚠️ 課包到期規則之合規免責： 「課包堂數自購買日起 30 天到期、逾期失效且不予退費」之規則及其商業設計，由客戶決定並自行確認其於目標市場（含美國聯邦 CARD Act 及各州，如加州、華盛頓州關於儲值金、禮券／gift card 到期之規範，及消費者保護法令）之合法性與合規性。開發方僅依客戶確認之規則進行技術實作，不就該規則之合法性、「到期不退」設計所生之任何爭議、罰款或集體訴訟風險負任何責任。客戶確認採用本規則，即視為已自行審查並承擔相關合規責任。
